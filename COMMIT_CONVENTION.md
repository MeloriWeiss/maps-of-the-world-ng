# Правила оформления коммитов

Этот проект использует [Conventional Commits](https://www.conventionalcommits.org/) для стандартизации сообщений коммитов.

## Формат коммита

```
<тип>(<область>): <описание>

[необязательное тело]

[необязательный футер]
```

### Примеры

```bash
feat(frontend): добавить компонент UserCard
fix(backend): исправить обработку ошибок в API
docs: обновить README с инструкциями по установке
style(frontend): отформатировать код согласно prettier
refactor(backend): упростить логику валидации
test(frontend): добавить тесты для Button компонента
```

## Типы коммитов

Должны поддерживаться следующие типы:

- `feat` - новая функциональность
- `fix` - исправление бага
- `docs` - изменения в документации
- `style` - форматирование кода
- `refactor` - рефакторинг
- `perf` - оптимизация производительности
- `test` - добавление тестов
- `build` - изменения в сборке
- `ci` - изменения в CI/CD
- `chore` - рутинные задачи
- `revert` - откат изменений

### `feat` - Новая функциональность

Добавление новой функции или возможности.

```bash
feat(frontend): добавить форму регистрации
feat(backend): реализовать API для загрузки файлов
```

### `fix` - Исправление бага

Исправление ошибки в коде.

```bash
fix(frontend): исправить отображение модального окна
fix(backend): устранить утечку памяти в обработчике
```

### `docs` - Документация

Изменения только в документации.

```bash
docs: добавить примеры использования API
docs(readme): обновить инструкции по развертыванию
```

### `style` - Стиль кода

Изменения, не влияющие на логику (форматирование, пробелы, точки с запятой).

```bash
style: применить prettier ко всем файлам
style(frontend): исправить отступы в компонентах
```

### `refactor` - Рефакторинг

Изменение кода без добавления функций или исправления багов.

```bash
refactor(backend): извлечь общую логику в утилиты
refactor(frontend): упростить структуру компонента
```

### `perf` - Производительность

Изменения, улучшающие производительность.

```bash
perf(frontend): оптимизировать рендеринг списка
perf(backend): добавить кэширование запросов
```

### `test` - Тесты

Добавление или изменение тестов.

```bash
test(frontend): добавить тесты для хука useAuth
test(backend): покрыть тестами API эндпоинты
```

### `build` - Сборка

Изменения в системе сборки или зависимостях.

```bash
build: обновить webpack до версии 5
build(deps): обновить react до 18.3
```

### `ci` - CI/CD

Изменения в конфигурации CI.

```bash
ci: добавить GitHub Actions для тестов
ci: настроить автоматический деплой
```

### `chore` - Рутинные задачи

Другие изменения, не затрагивающие src или test файлы.

```bash
chore: обновить .gitignore
chore: добавить prettier конфигурацию
```

### `revert` - Откат

Откат предыдущего коммита.

```bash
revert: откатить "feat: добавить новую функцию"
```

## Область (scope)

Область указывает, какая часть проекта затронута:

- `web` - изменения во frontend части
- `api` - изменения в backend части
- `deps` - изменения зависимостей
- `config` - изменения конфигурации

Область необязательна, но рекомендуется.

## Описание (subject)

- Используйте повелительное наклонение: "добавить" вместо "добавил"
- Не ставьте точку в конце
- Максимум 100 символов
- Начинайте с маленькой буквы (после двоеточия)

### ✅ Хорошо

```bash
feat: добавить валидацию email
fix: исправить ошибку при загрузке
```

### ❌ Плохо

```bash
feat: Добавил валидацию email.
fix: исправление ошибки
```

## Тело коммита (body)

Необязательное подробное описание изменений:

```bash
feat(frontend): добавить пагинацию для списка пользователей

Реализована пагинация с возможностью выбора количества элементов на странице.
Добавлены компоненты Pagination и PageSizeSelector.
Обновлен API клиент для поддержки параметров пагинации.
```

## Футер (footer)

Используется для ссылок на issue или breaking changes:

```bash
fix(backend): исправить обработку ошибок

Closes #123
```

```bash
feat(api): изменить формат ответа API

BREAKING CHANGE: формат ответа изменен с массива на объект с полями data и meta
```

## Автоматические проверки

При коммите автоматически запускаются:

1. **lint-staged** - проверка и форматирование измененных файлов
   - ESLint для проверки кода
   - Prettier для форматирования
2. **test** - тестирование всего проекта
3. **commitlint** - проверка формата сообщения коммита

### Если коммит отклонен

```bash
# Пример ошибки
⧗   input: добавил новую функцию
✖   subject may not be empty [subject-empty]
✖   type may not be empty [type-empty]
```

Исправьте сообщение коммита:

```bash
git commit --amend -m "feat: добавить новую функцию"
```

## Примеры хороших коммитов

```bash
# Простой коммит
feat(frontend): добавить кнопку выхода

# С телом
fix(backend): исправить утечку памяти в WebSocket

Проблема возникала при закрытии соединения.
Добавлена правильная очистка обработчиков событий.

# С breaking change
feat(api): изменить структуру ответа пользователя

BREAKING CHANGE: поле 'name' разделено на 'firstName' и 'lastName'

# С закрытием issue
fix(frontend): исправить баг с модальным окном

Модальное окно не закрывалось при клике вне его области.

Closes #456

# Множественные изменения
refactor(backend): улучшить структуру проекта

- Разделить роуты на отдельные файлы
- Добавить middleware для валидации
- Улучшить обработку ошибок
```
